## What This Builds

A compact moderation workflow:

- `/staff` toggles staff mode.
- `/freeze <player> [reason]` freezes or unfreezes a player.
- `/inspect <player>` opens a read-only info menu.
- `/staffchat <message>` sends to permitted online staff.
- Frozen players cannot move between blocks.
- Staff actions use sound/particle feedback and cleanup on quit.

## Lifecycle

- Staff mode and freeze state are stored by UUID.
- `PlayerMoveEvent` checks freeze state and cancels meaningful movement.
- `PlayerQuitEvent` removes transient session state.
- `sculk.close()` unregisters listeners and commands.

## Implementation Walkthrough

```kotlin
sculk.events.listen<PlayerMoveEvent> { event ->
    if (!staff.shouldBlockMovement(event.player.uniqueId)) return@listen
    if (event.from.blockX != event.to.blockX || event.from.blockZ != event.to.blockZ) {
        event.isCancelled = true
    }
}
```

The service is pure and testable. The plugin layer handles Bukkit players, events, menus, and effects.

## Threading Rules

- Event handlers run sync and should stay lightweight.
- Do not perform storage writes inside movement handlers.
- Log important moderation actions, but keep heavy audit storage async.

## Failure Behavior

- Freeze toggles off if the player is already frozen.
- Staff chat filters recipients by permission.
- Quit cleanup removes transient state for the quitting UUID.

## Common Mistakes

- Tracking staff state by player name.
- Doing database work in `PlayerMoveEvent`.
- Forgetting to unregister listeners.
- Mixing vanish or packet behavior into staff mode before packet support is implemented.
