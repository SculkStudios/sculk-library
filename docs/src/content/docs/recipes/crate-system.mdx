## What This Builds

A usable crate workflow:

- Config-defined crates and rewards.
- Key items marked with persistent data.
- `/crate preview <crate>` GUI.
- `/crate key give <player> <crate> <amount>` admin flow.
- `/crate open <crate>` key consumption and reward roll.
- Effects and broadcasts for wins.

## Commands

| Command | Purpose |
|---|---|
| `/crate preview <crate>` | Show reward preview. |
| `/crate open <crate>` | Consume a key and roll. |
| `/crate key give <player> <crate> <amount>` | Give crate keys. |
| `/crate reload` | Reload crate config. |

## Permissions

| Permission | Used By |
|---|---|
| `crate.admin` | Key give and reload. |

## Config

```kotlin
@ConfigFile("crates.yml")
data class CrateSettings(
    val crates: Map<String, CrateDefinition> = mapOf(...)
)

data class CrateDefinition(
    val displayName: String,
    val keyItem: ItemDescriptor,
    val rewards: List<CrateReward>,
)
```

Rewards use `ItemDescriptor`, so the same item config can be displayed in GUIs and delivered to players.

## Data Model

The crate recipe is config-driven. It does not store open history. Add a repository later if you want audit logs or pity counters.

## Lifecycle

- Load and validate config on enable.
- Build `CrateService` from current config.
- Reload replaces the service and menu factory.
- Close Sculk on disable.

## Implementation Walkthrough

Key items are marked with PDC:

```kotlin
crate.keyItem.copy(
    data = crate.keyItem.data + ("crate_id" to crateId)
)
```

Opening a crate:

1. Check inventory for a matching key marker.
2. Consume one key.
3. Roll a weighted reward.
4. Convert the reward `ItemDescriptor` to an `ItemStack`.
5. Add to inventory or drop leftovers.
6. Play feedback and broadcast rare wins.

## Threading Rules

- Inventory and world drops are sync operations.
- Config reload is small and sync in this example.
- If you add persistent crate logs, write them async.

## Failure Behavior

- Unknown crate IDs return a clear failure.
- Invalid reward weights are reported during validation.
- Invalid reward items are rejected when opened.
- Full inventories drop leftovers naturally at the player's location.

## Common Mistakes

- Giving rewards from preview clicks.
- Trusting config keys without validation.
- Forgetting to mark keys with PDC.
- Consuming a key before validating the crate exists.
- Blocking animation or audit storage on the main thread.
