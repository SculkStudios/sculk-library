title: GUIs Overview
description: Chest inventory GUIs with per-player sessions, click routing, and automatic cleanup.

GUIs are **immutable definitions**. Define once, open for any player. Sessions are created automatically on `openFor` and cleaned up when the player closes the inventory or disconnects through the platform GUI subsystem.

## Defining a GUI

```kotlin
// gui/MainMenu.kt
val mainMenu = gui("<dark_aqua><bold>Main Menu") {
    size = 27

    item(13) {
        material = Material.NETHER_STAR
        name = "<gold><bold>Click Me"
        lore(
            "<gray>Does something cool.",
            "<dark_gray>Right-click for more options.",
        )
        onClick {
            reply("<green>Clicked!")
            close()
        }
    }
}
```

## Opening for a player

```kotlin
// inside a command handler
player {
    mainMenu.openFor(player!!)
}
```

## Click context

Inside `onClick { }`, `this` is a `GuiContext`:

| Property / method | Description |
| --- | --- |
| `player` | The player who clicked |
| `clickType` | Click type (`LEFT`, `RIGHT`, `SHIFT_LEFT`, …) |
| `event` | Raw `InventoryClickEvent` — already cancelled |
| `session` | The active `GuiSession` — access state, refresh slots, update entries |
| `reply(msg)` | Send a MiniMessage message to the player |
| `close()` | Close the inventory |
| `open(gui)` | Switch to another GUI without reopening manually |

## Switching GUIs

```kotlin
onClick {
    open(settingsMenu)  // seamless switch — no flicker
}
```

## Open and close handlers

Use lifecycle handlers for per-session setup or cleanup. The handlers receive the active `GuiSession`.

```kotlin
val menu = gui("<dark_gray>Profile") {
    size = 27

    onOpen { session ->
        session.state["openedAt"] = System.currentTimeMillis()
    }

    onClose { session ->
        profileViewTracker.closed(session.player.uniqueId)
    }

    item(13) {
        material = Material.PLAYER_HEAD
        name("<aqua>Your Profile")
    }
}
```

Close handlers run from the platform GUI listener. Keep heavy database or file work off the main thread; schedule it through `sculk.scheduler.runAsync { ... }`.

## Dynamic items

Items whose content differs per player — use `dynamicContent { player -> }`. The block runs when the GUI opens, so the displayed stack is always up to date:

```kotlin
item(4) {
    material = Material.PLAYER_HEAD
    dynamicContent { player ->
        name = "<aqua>Welcome, <white>${player.name}"
        lore("<gray>Coins: <gold>${economy.getBalance(player)}")
    }
}
```

## Filling and borders

Two helpers cover the most common decoration patterns — filling all empty slots with filler glass, or just the outer ring:

```kotlin
gui("<dark_aqua>Shop") {
    size = 54

    // Fill the entire outer border with grey glass
    border(Material.GRAY_STAINED_GLASS_PANE) { name = "<gray> " }

    // Fill every remaining empty slot with black glass
    fill(Material.BLACK_STAINED_GLASS_PANE) { name = "<gray> " }

    // Your actual content slots go here — fill/border skip assigned slots
    item(22) { material = Material.DIAMOND; name = "<aqua>Diamonds!" }
}
```

Both methods accept an optional `GuiItemBuilder` block so you can set the name, lore, or click handler on the filler items. `fill()` and `border()` only ever affect **unassigned** slots — they will never overwrite an `item()` you've already defined, regardless of call order.

For premium menu layouts that only decorate the top and bottom rows, use `GuiSlots` and the styled border overload:

```kotlin
import studio.sculk.core.gui.GuiBorderStyle

gui("<dark_gray>Rewards") {
    size = 36

    border(
        material = Material.LIGHT_BLUE_STAINED_GLASS_PANE,
        style = GuiBorderStyle.Horizontal,
        skip = setOf(27, 31),
    ) {
        name = "<gray> "
    }

    item(27) { material = Material.ARROW; name = "<gray>Back" }
    item(31) { material = Material.BARRIER; name = "<gray>Close" }
}
```

`GuiSlots.horizontalBorder(27)` returns slots `0..8` and `18..26`; `GuiSlots.horizontalBorder(36)` returns `0..8` and `27..35`. This avoids the common off-by-one mistake where a 3-row menu accidentally tries to write slot `27`.

### Setting multiple slots at once

Use `items(slots)` to apply the same builder to an arbitrary set of slots:

```kotlin
// Corner accents
items(listOf(0, 8, 45, 53)) {
    material = Material.NETHER_STAR
    name = "<gold>★"
}

// An entire row
items(36..44) {
    material = Material.WHITE_STAINED_GLASS_PANE
    name = "<gray> "
}
```

## Item glow, enchantments, and custom model data

Sculk writes item metadata through Paper 26.1.2-compatible APIs:

- `glow = true` uses Paper's enchantment glint override instead of adding a hidden fake enchantment.
- `customModelData = 1001` keeps the DSL simple while writing a modern `CustomModelDataComponent` internally.
- `enchantment("sharpness", 5)` resolves enchantments through Paper registry access.

Add an enchantment shimmer to an item without showing any enchantment text:

```kotlin
item(22) {
    material = Material.DIAMOND
    name = "<aqua>Selected"
    glow = true   // shimmer with no tooltip
}
```

Apply specific enchantments by Minecraft key — useful for kit display items:

```kotlin
item(13) {
    material = Material.DIAMOND_SWORD
    name = "<#A1D8F5>Fang Kit"
    enchantment("sharpness", 5)
    enchantment("unbreaking", 3)
    enchantment("fire_aspect", 2)
}
```

For resource-pack item overrides, set `customModelData`:

```kotlin
item(4) {
    material = Material.STICK
    name = "<gold>Custom Item"
    customModelData = 1001   // writes a single float into the custom model data component
}
```

## Session state

Every open GUI has a `GuiSession` with a `state` bag — an arbitrary key-value store for driving dynamic behaviour within the session.

```kotlin
// Opening the GUI — store initial state
val session = filterMenu.openFor(player)
session.state["category"] = "weapons"
session.setEntries(getItemsForCategory("weapons"))

// Inside an onClick handler — typed read avoids unsafe casts
onClick {
    val category = session.state.get<String>("category") ?: "all"
    session.state["category"] = "armour"
    session.setEntries(getItemsForCategory("armour"))
    session.refresh(4)   // re-render the category indicator slot
}
```

Use `session.state.get<T>("key")` for a safe typed read that returns `null` rather than throwing on a type mismatch. The untyped `session.state["key"]` returns `Any?` and requires a manual cast.

State is tied to the `GuiSession` — it is cleared automatically when the player closes the inventory or the session ends. Use `open(otherGui)` to switch menus; the session (and its state) persists across the switch.

## Best practices

- Define GUIs as top-level `val`s or `object` constants — created once, reused for every player.
- Keep `onClick` handlers short. Heavy logic belongs in a service.
- Use `open(other)` rather than calling `player.closeInventory()` directly — it preserves session state.
- Enable `gui()` when creating `SculkPlatform`; without it, click and close routing is not registered.
