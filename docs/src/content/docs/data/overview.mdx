`sculk-data` gives you typed persistent storage without writing routine SQL. You annotate a data class, create a repository, and call `find`, `save`, and `delete`. SQLite is the default; MySQL/MariaDB is a config switch away.

## Defining an entity

Annotate a data class with `@Table` and mark each field with `@Column`. The primary key gets `@PrimaryKey`:

```kotlin
@Table("player_data")
data class PlayerData(
    @PrimaryKey @Column("uuid") val uuid: UUID,
    @Column("coins") val coins: Long = 0,
    @Column("homes") val homes: Int = 0,
    @Column("play_time") val playTime: Long = 0,
)
```

Sculk reads the annotations during mapper setup and caches the mapping, so normal repository calls do not need to rediscover the entity shape each time.

## Creating a repository

```kotlin
val repo = sculk.data.repository<PlayerData, UUID>()
```

The repository creates the database table automatically if it does not exist. Columns present in the annotation but missing from the table are added with `ALTER TABLE`.

## Operations

All repository methods are **suspend functions** that run their JDBC work off-thread, and return
`SculkResult<T>` — never throw. Call them from a coroutine (`sculk.scope.launchAsync { ... }`):

```kotlin
// Find by primary key — returns SculkResult<PlayerData?>
val result = repo.find(player.uniqueId)
when (result) {
    is SculkResult.Success -> {
        val data = result.value ?: PlayerData(player.uniqueId)
        // use data
    }
    is SculkResult.Failure -> logger.warning("Find failed: ${result.message}")
}

// Save (insert or update via upsert)
repo.save(PlayerData(player.uniqueId, coins = 500, homes = 2))

// Delete by primary key
repo.delete(player.uniqueId)

// Check if a row exists
if (repo.exists(player.uniqueId)) { ... }

// Load all rows
val everyone: SculkResult<List<PlayerData>> = repo.findAll()
```

## Running in a coroutine

Repository methods suspend while the database work runs off-thread, so they never block the main
thread. Launch them on the platform scope and hop back with `withMain` to touch the Paper API:

```kotlin
sculk.scope.launchAsync {
    when (val result = repo.find(player.uniqueId)) {
        is SculkResult.Success -> {
            val coins = result.value?.coins ?: 0
            withMain { player.sendRichMessage("<yellow>You have $coins coins.") }
        }
        is SculkResult.Failure -> {
            withMain { player.sendRichMessage("<red>Failed to load your data.") }
        }
    }
}
```

See [Coroutines](/advanced/coroutines/) for the full scope/dispatcher model, and
[Querying, transactions & Redis](/data/query/) for the query DSL and distributed cache.

## Storage configuration

`sculk-data` generates a `storage.yml` in your plugin's data folder on first run:

```yaml
type: sqlite       # sqlite or mysql

sqlite:
  file: data.db    # relative to plugin data folder

mysql:
  host: localhost
  port: 3306
  database: sculk
  username: root
  password: ""
  pool-size: 10
```

Switch to MySQL by changing `type: mysql` and filling in the connection details. The schema migrations run automatically on startup for both backends.

## Multiple entities

Create one repository per entity type. Each gets its own table:

```kotlin
val playerRepo  = sculk.data.repository<PlayerData, UUID>()
val homeRepo    = sculk.data.repository<HomeData, Int>()
val economyRepo = sculk.data.repository<EconomyData, UUID>()
```

## Result chaining

`SculkResult` supports functional operators for transforming and handling results without nested `when` blocks:

```kotlin
// map — transform a successful value
val coins: SculkResult<Long> = repo.find(player.uniqueId)
    .map { data -> data?.coins ?: 0L }

// onSuccess / onFailure — side effects, returns the same result for chaining
repo.find(player.uniqueId)
    .onSuccess { data -> player.sendMessage("<yellow>Coins: ${data?.coins}") }
    .onFailure { message, _ -> logger.warning("DB error: $message") }

// fold — collapse to a single value
val display = repo.find(player.uniqueId).fold(
    onSuccess = { data -> "<yellow>Coins: ${data?.coins ?: 0}" },
    onFailure = { message, _ -> "<red>Error: $message" },
)
player.reply(display)

// recover — provide a fallback result on failure
val safe: SculkResult<PlayerData> = repo.find(player.uniqueId)
    .recover { _, _ -> PlayerData(player.uniqueId) }

// flatMap — chain an operation that itself returns SculkResult
// failures short-circuit: if find fails, the flatMap block never runs
repo.find(player.uniqueId)
    .flatMap { data ->
        if (data != null) SculkResult.success(data)
        else SculkResult.failure("player not found")
    }
    .onSuccess { data -> player.sendMessage("<green>Welcome back, ${data.name}!") }
    .onFailure { msg, _ -> player.sendMessage("<red>$msg") }
```

## Best practices

- Create repositories once at startup — don't create new instances per request.
- Call repository methods from `sculk.scope.launchAsync { }`; hop back with `withMain { }` for Paper APIs.
- Use the cache layer (`sculk.data.cached(...)`) for frequently-read entities like online player data. See [Caching](/data/caching).
- Handle `SculkResult.Failure` explicitly — log the message so operators can diagnose database issues.
- Load player data on `PlayerJoinEvent` and cache it; flush to DB on `PlayerQuitEvent`.
