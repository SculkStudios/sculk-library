Sculk Studio includes scheduler adapters for [Folia](https://papermc.io/software/folia) and its forks, including [Canvas MC](https://canvasmc.io). You still need to choose the right scheduler context for the work being done: async for IO, entity/location sync for Paper world and player APIs.

## What Folia changes

Folia replaces Paper's single main thread with per-region threads, one per group of nearby chunks. The old `BukkitScheduler` is removed. Four new schedulers take its place:

| Scheduler | Use case |
| --- | --- |
| **GlobalRegionScheduler** | Server-wide tasks with no entity/location context — equivalent to the old main thread |
| **EntityScheduler** | Tasks tied to a specific entity (e.g. opening a player's inventory) |
| **RegionScheduler** | Tasks tied to a specific world location (e.g. placing blocks) |
| **AsyncScheduler** | Background tasks off all region threads — database, HTTP, file I/O |

Sculk Studio detects Folia at startup via class presence and routes scheduler calls through the matching backend. Code that touches a player, entity, block, inventory, or world location should use the entity/location overloads so Folia can run it on the owning region thread.

## Required: declare Folia support

Folia will refuse to load plugins that don't opt in. Add one line to your `paper-plugin.yml`:

```yaml
name: MyPlugin
version: 1.0.0
main: com.example.MyPlugin
api-version: '1.21.11'
folia-supported: true   # required for Folia and Canvas
```

Without this flag your plugin will not start on Folia regardless of whether the code is safe.

## Scheduler — what changes

### `runSync` → global region thread

`sculk.scheduler.runSync { }` maps to `GlobalRegionScheduler` on Folia. This is the correct choice for tasks that don't touch a specific player or location.

### `runSync(entity)` and `runSync(location)` — new Folia-safe overloads

When your task operates on a specific player or world position, use the entity/location overloads. On Folia these dispatch to the correct region thread. On Paper they behave identically to `runSync`.

```kotlin
// After a DB query, update a player's inventory on their region thread
sculk.scope.launchAsync {
    val result = repo.find(player.uniqueId)   // suspends off-thread

    // runSync(player) — entity scheduler on Folia, global sync on Paper
    sculk.scheduler.runSync(player) {
        player.sendRichMessage("<yellow>Coins: ${result.getOrNull()?.coins}")
    }
}
```

### `runAsync` — unchanged

All async methods work the same on both platforms. Folia's `AsyncScheduler` uses real time internally; Sculk converts ticks automatically (`1 tick = 50 ms`).

## GUIs

`Gui.openFor(player)` auto-dispatches `player.openInventory()` to the entity scheduler on Folia. You call it the same way on both platforms:

```kotlin
player {
    val session = shopMenu.openFor(player!!)
    session.setEntries(shopItems())   // safe — operates on in-memory Inventory object
}
```

Prefer mutating GUI sessions from the same scheduler context you use to open the inventory. If the data comes from a database or file, load it with `runAsync`, then return to `runSync(player)` before calling `setEntries()`, `refresh()`, or `refreshAll()`.

## Canvas MC

Canvas is a Folia fork focused on stability and performance. It follows Folia's scheduler model, so the same Sculk entity/location scheduler guidance applies.

Canvas advantages you can use immediately in your plugins:
- Full Spark profiler support for region-threaded servers
- Faster chunk generation (transparent to plugins)
- Stability fixes for several Folia edge cases

## Summary

| What you need to do | What Sculk handles |
| --- | --- |
| Add `folia-supported: true` to `paper-plugin.yml` | Folia detection at startup |
| Use `scheduler.runSync(player)` for player-specific tasks from async | `Gui.openFor()` entity scheduler dispatch |
| Use async tasks for DB/file/HTTP work | Scheduler API routing (BukkitScheduler → Folia scheduler backend) |
| — | Tick-to-millisecond conversion for async delayed/repeating tasks |
