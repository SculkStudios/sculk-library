Sculk Studio keeps hot paths small, but it does not remove Paper's threading rules. Treat Paper APIs and database APIs deliberately.

## Main-thread rule

Paper player, world, inventory, and most entity APIs must run on the main thread or the correct Folia region/entity thread. Database, file, HTTP, and expensive CPU work should run off-thread.

Repository methods in `sculk-data` are synchronous and blocking. Run them inside async scheduler work, then return to sync before touching Paper APIs.

<Tabs>
<TabItem label="Kotlin">
```kotlin
sculk.scheduler.runAsync {
    val result = repo.find(player.uniqueId)

    sculk.scheduler.runSync(player) {
        player.sendMessage("Loaded: ${result.isSuccess}")
    }
}
```
</TabItem>
<TabItem label="Java">
```java
JavaRepository.wrap(repo)
    .find(player.getUniqueId())
    .thenAccept(result ->
        sculk.getScheduler().runSync(player, () ->
            player.sendMessage("Loaded: " + result.isSuccess())
        )
    );
```
</TabItem>
</Tabs>

## Reflection and mapping

`sculk-config` and `sculk-data` use reflection for data-class or entity mapping. Create config managers, repositories, and caches once during startup and reuse them. Do not create repositories per command execution or per player tick.

## Connection pooling

`sculk-data` uses HikariCP for JDBC connections.

| Backend | Default behavior |
| --- | --- |
| SQLite | Local file database, suitable for small plugins and local development. |
| MySQL/MariaDB | Network database, useful when multiple services need shared storage. |

Tune MySQL pool size only after measuring contention:

```yaml
mysql:
  pool-size: 20
```

## Cache behavior

`SculkCache` uses Caffeine. Cache hits avoid database access. Cache misses call the wrapped repository on the calling thread, so call cache-backed repository operations from async work too.

## GUI updates

`session.refresh(slot)` updates one slot in the current inventory. `session.refreshAll()` re-renders static slots and pagination. Keep click handlers short and move database or HTTP work into async scheduler tasks.

## Series lookup caching

`SculkSeries` normalizes and resolves a key on first use, then caches the result. Pre-warm important config keys during startup if you want missing-key warnings before gameplay.

```kotlin
override fun onEnable() {
    SculkSeries.material("diamond_sword")
    SculkSeries.sound("entity.player.levelup")
}
```

## Packet paths

Packet APIs are optional and split into separate modules. Do not install packet modules unless your plugin uses packet-powered features.

Packet listeners sit on hot networking paths:

- Prefer PacketEvents for new packet-heavy work.
- Keep ProtocolLib for compatibility with existing server stacks.
- Do not block packet callbacks.
- Do not touch Paper APIs directly from packet callbacks; use `context.runSync { ... }`.
- Use packet filters before decoding data.
- Keep debug listeners short-lived.

## Checklist

- Run repository and cache miss work asynchronously.
- Return to `runSync(player)` or `runSync(location)` before Paper API calls.
- Reuse repositories and caches.
- Keep GUI click handlers short.
- Pre-warm config-driven registry keys at startup.
- Install only packet modules your plugin uses.
- Measure before changing connection pool sizes.
