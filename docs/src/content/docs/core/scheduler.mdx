import { Tabs, TabItem } from '@astrojs/starlight/components';

`sculk.scheduler` wraps Paper's task scheduler. Every method returns a `SculkHandle` — call `close()` on it to cancel the task at any point.

## Threading rules

| Context | Paper API safe? | DB calls safe? |
| --- | --- | --- |
| Main thread (sync) | Yes | **No** — blocks TPS |
| Async thread | **No** | Yes |

Never call Paper APIs (entity, inventory, world) from an async task. Never call `SculkRepository` from the main thread.

## One-shot tasks

<Tabs>
<TabItem label="Kotlin">
```kotlin
// Run on the main thread next tick
sculk.scheduler.runSync {
    player.sendMessage("<green>Hello from next tick!")
}

// Run async (off the main thread)
sculk.scheduler.runAsync {
    val result = repo.find(player.uniqueId)
    sculk.scheduler.runSync {
        result.onSuccess { data ->
            player.sendMessage("Coins: ${data.coins}")
        }
    }
}
```
</TabItem>
<TabItem label="Java">
```java
sculk.getScheduler().runAsync(() -> {
    SculkResult<PlayerData> result = repo.find(player.getUniqueId());
    sculk.getScheduler().runSync(() ->
        result.onSuccess(data ->
            player.sendMessage("Coins: " + data.coins())
        )
    );
});
```
</TabItem>
</Tabs>

<Tabs>

```

```

## Repeating tasks

`runAsyncRepeating` is the correct choice for recurring DB operations (data flushes, heartbeats). The returned `SculkHandle` cancels the task when closed.

<Tabs>
<TabItem label="Kotlin">
```kotlin
// Flush dirty player data every 5 minutes (6000 ticks)
val flushHandle = sculk.scheduler.runAsyncRepeating(
    delayTicks  = 6000L,
    periodTicks = 6000L,
) {
    dirtyPlayers.forEach { uuid ->
        cache.find(uuid).onSuccess { data ->
            if (data != null) repo.save(data)
        }
    }
}

// Cancel when no longer needed
flushHandle.close()
```
</TabItem>
<TabItem label="Java">
```java
SculkHandle flushHandle = sculk.getScheduler().runAsyncRepeating(
    6000L, 6000L,
    () -> dirtyPlayers.forEach(uuid -> {
        cache.find(uuid).onSuccess(data -> {
            if (data != null) repo.save(data);
        });
    })
);

// Cancel
flushHandle.close();
```
</TabItem>
</Tabs>

For repeating sync tasks (e.g., visual effects on the main thread):

val handle = sculk.scheduler.runSyncRepeating(

SculkHandle handle = sculk.getScheduler().runSyncRepeating(0L, 10L, () ->

## Entity and location overloads (Folia / Canvas)

On Folia and Canvas, every chunk is owned by a specific region thread. Calling Paper APIs (inventory, entities) from the wrong thread causes errors. Use the entity/location overloads to dispatch to the correct thread automatically:

```kotlin
sculk.scheduler.runAsync {
    val result = repo.find(player.uniqueId)

    // runs on the thread owning the player's chunk — safe on both Paper and Folia
    sculk.scheduler.runSync(player) {
        player.sendMessage("Coins: ${result.getOrNull()?.coins}")
    }
}
```

On Paper, `runSync(entity)` and `runSync(location)` behave identically to `runSync` — no performance difference, no code change needed when switching between platforms.

See [Folia & Canvas](/advanced/folia) for the full compatibility guide.

## All scheduler methods

| Method | Thread | Fires |
| --- | --- | --- |
| `runSync(task)` | Main / global region | Next tick |
| `runSync(entity, task)` | Entity's region thread | Next tick |
| `runSync(location, task)` | Location's region thread | Next tick |
| `runSyncDelayed(delay, task)` | Main / global region | After `delay` ticks |
| `runSyncDelayed(entity, delay, task)` | Entity's region thread | After `delay` ticks |
| `runSyncRepeating(delay, period, task)` | Main / global region | Every `period` ticks |
| `runAsync(task)` | Async | Immediately (off main) |
| `runAsyncDelayed(delay, task)` | Async | After `delay` ticks |
| `runAsyncRepeating(delay, period, task)` | Async | Every `period` ticks |

20 ticks = 1 second at normal TPS.

## Cancellation

Every method returns a `SculkHandle`. Closing it cancels the task:

```kotlin
val handle = sculk.scheduler.runSyncRepeating(0L, 20L) { /* ... */ }

// Cancel after 10 seconds
sculk.scheduler.runSyncDelayed(200L) { handle.close() }
```

Tasks registered through `sculk.scheduler` are **not** auto-cancelled on `sculk.close()` — cancel them explicitly, or use `sculk.register(handle)` to have them cleaned up automatically:

```kotlin
sculk.register(
    sculk.scheduler.runAsyncRepeating(6000L, 6000L) { /* flush */ }
)
// Now automatically cancelled when sculk.close() is called
```
