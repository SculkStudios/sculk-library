## Module dependency flow

```text
sculk-core
  -> sculk-config
  -> sculk-series
  -> sculk-effects
  -> sculk-data
  -> sculk-platform
```

`sculk-platform` is the module that owns Paper plugin lifecycle integration. The lower modules expose reusable APIs that are easier to test and reason about without a full server runtime.

## Module roles

**`sculk-core`** contains the command DSL, GUI model, Adventure messaging helpers, `SculkResult`, scheduler contracts, and version parsing.

**`sculk-config`** maps YAML files to Kotlin data classes. It writes defaults, supports comments, validates annotated constructor parameters, and exposes reload callbacks.

**`sculk-series`** provides registry-backed lookups for modern Paper keys. It normalizes key input and returns `null` for missing values. It is not a complete historical alias database.

**`sculk-effects`** wraps particle and sound work in readable builders and provides timelines/sequences for scheduled effects.

**`sculk-data`** provides JDBC repositories and cache helpers. Repository methods are synchronous and blocking, so call them from async scheduler work.

**`sculk-platform`** wires the library into a `JavaPlugin`: commands, events, scheduler, config, data, and GUI lifecycle cleanup.

## Lifecycle model

Resources that need cleanup implement `SculkHandle`. Examples include listener registrations, scheduled tasks, platform services, and data pools.

`SculkPlatform.close()` closes platform-owned handles, GUI sessions, commands, event listeners, and data resources. Scheduled tasks returned directly from `sculk.scheduler` should be retained and closed by your plugin unless you register them in your own lifecycle owner.

```kotlin
class MyPlugin : JavaPlugin() {
    private lateinit var sculk: SculkPlatform

    override fun onEnable() {
        sculk = SculkPlatform.create(this) {
            gui()
            config()
            data()
        }

        sculk.commands.registerAll(
            homeCommand(sculk),
            adminCommand(sculk),
        )
    }

    override fun onDisable() {
        sculk.close()
    }
}
```

## Threading model

- Use sync scheduler methods for Paper player, world, inventory, and event-facing work.
- Use async scheduler methods for JDBC, file IO, HTTP, and CPU-heavy work.
- On Folia, use entity/location-aware sync overloads when the work touches a specific entity or location.
- Repository methods are blocking; the library does not make them automatically safe to call on the main thread.

## Error handling

Operations that can fail commonly return `SculkResult<T>` instead of throwing. Handle both success and failure paths explicitly.

```kotlin
when (val result = repo.find(player.uniqueId)) {
    is SculkResult.Success -> player.sendMessage("Coins: ${result.value?.coins ?: 0}")
    is SculkResult.Failure -> logger.warning("Load failed: ${result.message}")
}
```

## API stability markers

| Annotation | Meaning |
| --- | --- |
| `@SculkStable` | Intended for normal use and protected by the release policy. |
| `@SculkExperimental` | Available, but the shape may change while feedback is gathered. |
| `@SculkInternal` | Framework internals with no compatibility guarantee. |
