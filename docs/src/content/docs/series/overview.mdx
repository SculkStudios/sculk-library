`sculk-series` resolves string keys into Paper/Bukkit registry values. It is useful when keys come from config files, messages, command arguments, or data stores.

It is intentionally narrower than XSeries. It does not ship a full historical alias database for every renamed material, sound, potion, or particle. Unknown keys return `null` so your plugin can choose a fallback or disable a feature cleanly.

## What it supports

- Paper registry-backed lookups for sounds, particles, entity types, enchantments, potion effects, and biomes.
- Material lookup through Paper/Bukkit material matching.
- Case-insensitive input where the backing API supports it.
- `minecraft:` namespace handling for registry keys.
- Dot/underscore normalization for common config input styles.
- Cached lookup results through `SculkRegistry`.

## Null-safe fallback

Always handle missing keys. A key can be absent because the server version does not contain it, the user misspelled it, or a config was copied from another plugin.

```kotlin
val material =
    SculkSeries.material(config.iconMaterial)
        ?: run {
            logger.warning("Unknown material '${config.iconMaterial}', using STONE.")
            Material.STONE
        }
```

## When to use it

Use `SculkSeries` for external input:

- Materials in config files.
- Sounds in message files.
- Particle names in effect definitions.
- Admin command arguments.
- Stored registry keys in a database.

For hardcoded plugin logic, direct Paper constants are often clearer.

## XSeries comparison

`sculk-series` is not a direct XSeries clone. It focuses on modern Paper registry access, clean nullable lookups, `require*` helpers, startup validation reports, and a small set of curated aliases for common config migrations.
