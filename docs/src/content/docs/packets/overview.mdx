---
title: Packets
description: Optional packet APIs for client-side features without making core heavy.
---

Sculk packet support is deliberately split into small optional modules. The core modules and `sculk-platform` do not bundle PacketEvents or ProtocolLib, and Sculk does not shade either packet library.

Use packets when you need client-side behavior that Paper does not expose directly. Today, Sculk exposes production APIs for low-level listeners, packet sending, debug packet sessions, and temporary client-side block previews. Prefer Paper APIs when they already solve the job.

## Modules

| Module | Purpose |
| --- | --- |
| `sculk-packets-api` | Small backend-neutral contracts and high-level packet service APIs. |
| `sculk-packets-packetevents` | Optional PacketEvents adapter. Recommended for new packet-heavy plugins. |
| `sculk-packets-protocollib` | Optional ProtocolLib adapter for servers already using ProtocolLib. |

## Install only what you need

You already have the packet **API** through `sculk-platform`. To turn packets on, add **one backend
adapter** — it pulls the packet API in transitively. Pick the one matching the packet library on your
server:

PacketEvents path:

```kotlin
dependencies {
    implementation("com.github.SculkStudios.sculk-library:sculk-platform:4.0.0")
    implementation("com.github.SculkStudios.sculk-library:sculk-packets-packetevents:4.0.0")
}
```

ProtocolLib path:

```kotlin
dependencies {
    implementation("com.github.SculkStudios.sculk-library:sculk-platform:4.0.0")
    implementation("com.github.SculkStudios.sculk-library:sculk-packets-protocollib:4.0.0")
}
```

The PacketEvents/ProtocolLib libraries themselves are `compileOnly` — install the matching plugin on
your server (and add a `softdepend`).

You must also install PacketEvents or ProtocolLib on the server. Sculk only provides adapters.

## Enable packets

```kotlin
import studio.sculk.packets.PacketBackendMode

class MyPlugin : SculkPlugin({
    packets {
        backend = PacketBackendMode.Auto
        required = false
    }
}) {
    override fun setup() {
        sculk.packetsResult
            ?.onSuccess { service -> logger.info("Packets enabled with ${service.backend}") }
            ?.onFailure { message, _ -> logger.warning(message) }
    }
}
```

`Auto` prefers PacketEvents, then ProtocolLib. If `required = true`, startup fails with a clear message when no backend is available.

## Backend status

`sculk-packets-api` provides the shared contracts and high-level service surface. The PacketEvents adapter is the preferred backend for new packet-heavy plugins and supports low-level listener registration, cancellation, changed-packet marking, listener cleanup, and sending `PacketEventsPacket` wrappers. The ProtocolLib adapter remains available for servers that already standardize on ProtocolLib and supports listener registration, cleanup, and sending `ProtocolLibPacket` containers.

## Current high-level API

Client-side blocks are available through the backend-neutral service:

```kotlin
val packets = sculk.packets

packets.clientBlocks.set(player, targetBlock.location, Material.DIAMOND_BLOCK)
packets.clientBlocks.preview(player, targetBlock.location, Material.EMERALD_BLOCK, durationTicks = 60)
packets.clientBlocks.reset(player, targetBlock.location)
```

The public packet contracts also include debug sessions, low-level listener registration, and packet sending. `sculk-content` currently keeps the higher-level client block facade separate from the packet contracts; backend-specific packet wrappers remain isolated to adapter modules so the core jar stays small.

## Threading

Packet listeners can run off the main thread depending on the backend and packet type. Do not touch Paper player, world, inventory, or entity APIs directly from a packet callback. Use `context.runSync { ... }` before interacting with Paper state.

## Common mistakes

- Adding both adapters but not installing PacketEvents or ProtocolLib on the server.
- Shading PacketEvents or ProtocolLib into your plugin. Install the runtime plugin instead.
- Doing database or file IO inside a packet listener.
- Modifying a PacketEvents wrapper and forgetting to mark it changed.
- Using packets for behavior Paper already supports safely.
