title: Commands Overview
description: DSL-first command system with sender routing, typed arguments, and tab-completion from the command tree.
---

Commands are defined with the `command {}` DSL and registered via `sculk.commands.registerAll()`.
Each command lives in its own file and returns a `CommandBuilder`.

## Defining a command

```kotlin
// commands/AdminCommand.kt
fun adminCommand() = command("sculk") {
    description = "Sculk admin commands."
    permission = "sculk.admin"

    sub("reload") {
        player {
            sculk.config.reload()
            reply("<green>Config reloaded.")
        }
    }

    sub("ping") {
        executes {
            reply("<gray>Pong!")
        }
    }
}
```

## Registering commands

```kotlin
sculk.commands.registerAll(
    adminCommand(),
    homeCommand(),
    shopCommand(),
)
```

## Sender-type routing

| Block | Who runs it | Rejects |
| --- | --- | --- |
| `player { }` | Online player | Console (auto error) |
| `console { }` | Server console | Players (auto error) |
| `executes { }` | Anyone | — |

`player` is non-null inside a `player { }` block — no null checks needed.

## Aliases

Commands can have aliases — additional names that trigger the same handler:

```kotlin
fun teleportCommand() = command("teleport") {
    aliases = listOf("tp", "tele")
    player {
        // /teleport, /tp, and /tele all work
    }
}
```

## Auto tab-completion

Generated from the subcommand tree and registered argument names. Add explicit suggestions on arguments when a command needs dynamic values such as online players, worlds, or config keys.

## Permission checks

Set `permission` on any node. Players without the permission get a clean error message automatically.
Root-level permissions cascade — a player denied at the root never reaches subcommands.

Sculk checks node permissions during command dispatch. The Bukkit/Paper command permission is still set for command visibility and registration behavior, but Sculk does not rely on deprecated Bukkit permission-message handling for normal player feedback.

:::note
Commands currently register through Bukkit's command map. That keeps the DSL small and predictable while Sculk continues to target modern Paper. If you need Brigadier-specific client UX today, use Paper's command API directly for that command and keep Sculk commands for the rest of your plugin.
:::

## Feedback helpers

Inside any executor block, `this` is a `CommandContext` with helpers beyond `reply()`:

```kotlin
player {
    reply("<green>Operation complete.")        // chat message
    actionbar("<gray>Processing…")             // above hotbar
    title("<aqua>Welcome!", "<gray>to the server")  // full-screen title
    playSound(Sound.ENTITY_PLAYER_LEVELUP)     // sound at player location
}
```

`title` and `actionbar` are no-ops when the sender is console. `playSound` defaults to `volume = 1.0, pitch = 1.0` — pass extra args to override.

## Best practices

- One command per file, returning a `CommandBuilder` from a top-level function.
- Register everything in `onEnable` with `registerAll`.
- Use `executes { }` when behaviour is identical for players and console.
- Keep handlers thin — delegate heavy logic to a service class.
