Commands are defined with the `command {}` DSL (`SculkCommands.command(...)` in Java) and registered
via `sculk.commands.registerAll()`. Each command lives in its own file and returns a `CommandBuilder`.

> **Java note:** executors (`player`/`console`/`executes`) are `suspend` on the Kotlin side, which Java
> cannot target — so the `Consumer` overloads are the only ones Java sees. No ambiguity, even for
> one-liners.

## Defining a command

<Tabs syncKey="lang">
<TabItem label="Kotlin">

```kotlin
// commands/AdminCommand.kt
fun adminCommand() = command("sculk") {
    description = "Sculk admin commands."
    permission = "sculk.admin"

    sub("reload") {
        player {
            sculk.config.reloadAll()
            reply("<green>Config reloaded.")
        }
    }

    sub("ping") {
        executes {
            reply("<gray>Pong!")
        }
    }
}
```

</TabItem>
<TabItem label="Java">

```java
// commands/AdminCommand.java
CommandBuilder adminCommand() {
    return SculkCommands.command("sculk", cmd -> {
        cmd.setDescription("Sculk admin commands.");
        cmd.setPermission("sculk.admin");

        cmd.sub("reload", sub -> sub.player(ctx -> {
            getSculk().getConfig().reloadAll();
            ctx.reply("<green>Config reloaded.");
        }));

        cmd.sub("ping", sub -> sub.executes(ctx -> {
            ctx.reply("<gray>Pong!");
        }));
    });
}
```

</TabItem>
</Tabs>

## Sender-type routing

| Block | Who runs it | Rejects |
| --- | --- | --- |
| `player { }` | Online player | Console (auto error) |
| `console { }` | Server console | Players (auto error) |
| `executes { }` | Anyone | — |

`player` is non-null inside a `player { }` block — no null checks needed (`ctx.getPlayer()` in Java).

## Aliases

Commands can have aliases — additional names that trigger the same handler:

<Tabs syncKey="lang">
<TabItem label="Kotlin">

```kotlin
fun teleportCommand() = command("teleport") {
    aliases = listOf("tp", "tele")
    player {
        // /teleport, /tp, and /tele all work
    }
}
```

</TabItem>
<TabItem label="Java">

```java
CommandBuilder teleportCommand() {
    return SculkCommands.command("teleport", cmd -> {
        cmd.setAliases(java.util.List.of("tp", "tele"));
        cmd.player(ctx -> {
            // /teleport, /tp, and /tele all work
        });
    });
}
```

</TabItem>
</Tabs>

## Auto tab-completion

Generated from the subcommand tree and registered argument names. Add explicit suggestions on arguments when a command needs dynamic values such as online players, worlds, or config keys.

## Permission checks

Set `permission` on any node. Players without the permission get a clean error message automatically.
Root-level permissions cascade — a player denied at the root never reaches subcommands.

## Feedback helpers

Inside any executor block the context (`this` in Kotlin, `ctx` in Java) is a `CommandContext` with
helpers beyond `reply()`:

<Tabs syncKey="lang">
<TabItem label="Kotlin">

```kotlin
player {
    reply("<green>Operation complete.")        // chat message
    actionbar("<gray>Processing…")             // above hotbar
    title("<aqua>Welcome!", "<gray>to the server")  // full-screen title
    playSound(Sound.ENTITY_PLAYER_LEVELUP)     // sound at player location
}
```

</TabItem>
<TabItem label="Java">

```java
cmd.player(ctx -> {
    ctx.reply("<green>Operation complete.");        // chat message
    ctx.actionbar("<gray>Processing…");             // above hotbar
    ctx.title("<aqua>Welcome!", "<gray>to the server");  // full-screen title
    ctx.playSound(Sound.ENTITY_PLAYER_LEVELUP);     // sound at player location
});
```

</TabItem>
</Tabs>

`title` and `actionbar` are no-ops when the sender is console. `playSound` defaults to `volume = 1.0, pitch = 1.0`.

## Best practices

- One command per file, returning a `CommandBuilder`.
- Register everything in `setup()` with `registerAll`.
- Use `executes { }` when behaviour is identical for players and console.
- Keep handlers thin — delegate heavy logic to a service class.
